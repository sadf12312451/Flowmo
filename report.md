# 浮墨 Flowmo 智能生活管理桌面应用项目报告

## 一、引言

### 摘要

浮墨 Flowmo 是一款面向普通用户的 AI 桌面生活管理应用，围绕便签、待办、倒计时、番茄钟、习惯打卡、日历聚合和全局搜索等日常事务，构建“能理解、能记录、能提醒、能执行”的个人生活管家。与传统效率工具相比，浮墨不再要求用户在多个应用之间手动拆分任务，而是通过自然语言交互，将用户表达自动转化为待办、提醒、便签或习惯计划。系统采用 Tauri 2、Vue 3、TypeScript、SQLite 等技术实现，兼顾桌面端性能、跨平台能力、本地数据存储与隐私保护。

### 1. 背景

当前个人效率工具数量众多，但普遍存在三类不足。第一，传统待办、便签、日历、番茄钟工具功能分散，用户需要在多个应用之间切换，增加了管理成本。第二，通用 AI 助手虽然具备理解能力，但多数停留在“回答建议”层面，无法直接完成具体生活事务的创建、更新和提醒。第三，云端 AI 与在线效率工具高度依赖账号、网络和远程数据处理，容易引发隐私顾虑。

随着生成式 AI 被广泛使用，用户已经习惯用自然语言描述需求。Microsoft 与 LinkedIn 的 2024 工作趋势报告显示，75% 的知识工作者已经在工作中使用 AI[1]。但 Pew Research Center 的调查也指出，70% 听说过 AI 的美国成年人对企业负责任使用 AI 的信任度较低，81% 担心个人信息被用于不舒适的场景[2]。这说明市场既需要 AI 能力，也需要更可控、更贴近日常生活场景的产品形态。

### 2. 优势

浮墨针对上述痛点形成了四点优势。

第一，从“AI 回答”升级为“AI 执行”。用户说出“明天 8 点提醒我吃药”，系统可直接生成提醒事项，而不是只给出口头回应。

第二，从单点工具升级为生活事务中枢。便签、待办、倒计时、番茄钟、习惯、日历和搜索被统一组织，减少用户在多个工具之间切换。

第三，从云端依赖升级为本地优先。核心生活数据存储在本地 SQLite 中，降低隐私泄露风险，也增强了用户对个人数据的控制感。

第四，从专业效率工具升级为普通人友好界面。产品弱化复杂配置和技术术语，强调即时反馈、桌面常驻、悬浮便签和低学习成本。

### 3. 怎么做的

项目采用“桌面端壳层 + 前端交互层 + 本地数据层 + AI 意图层”的整体路径。Tauri 负责桌面窗口、系统托盘、悬浮窗和原生能力调用；Vue 3 与 Naive UI 构建用户界面；Pinia 负责模块状态管理；SQLite 负责本地持久化；AI 意图层将自然语言解析为结构化操作，再分发到待办、便签、倒计时等模块执行，形成“用户表达 - 系统理解 - 自动执行 - 结果反馈”的闭环。

## 二、文献综述 / 事实和证据

### 1. 同类方案不足的事实支撑

| 现有方案 | 主要特点 | 存在不足 | 对本项目的启示 |
| --- | --- | --- | --- |
| 传统待办、便签、日历工具 | 单一功能成熟 | 数据割裂、切换成本高、依赖手动录入 | 需要统一生活事务入口 |
| 通用 AI 聊天助手 | 理解能力强 | 多数场景无法直接落地到用户本地任务系统 | 需要“理解 + 执行”闭环 |
| 云端效率工具 | 多端同步方便 | 用户对个人数据和 AI 数据使用存在担忧 | 需要本地优先和隐私可控 |
| 普通提醒工具 | 可按时通知 | 提醒时机不当可能造成打扰 | 需要更智能、更克制的提醒策略 |

提醒系统的价值已有研究支撑。Ferguson 等人的研究表明，智能手机提醒可以提升存在前瞻记忆困难用户的日常任务完成表现[3]。但通知系统也可能带来干扰，智能通知系统综述指出，在不合适时机出现的通知可能造成中断和负面体验[4]。因此，浮墨不是简单增加提醒数量，而是强调任务上下文、优先级、提前提醒和用户可控。

### 2. 项目设计逻辑的理论支撑

技术接受模型指出，用户是否接受一个信息系统，核心受“感知有用性”和“感知易用性”影响[5]。浮墨的产品逻辑正是围绕这两点展开：通过 AI 自动创建任务提升有用性，通过自然语言入口和桌面常驻降低使用门槛。

本地优先软件理念强调用户应拥有自己的数据，即使存在云服务，也不应让用户失去数据控制权[6]。浮墨采用本地 SQLite 存储，符合本地优先理念。Tauri 官方文档说明其支持以 Web 前端和 Rust 后端构建小体积、安全、跨平台应用[7]，SQL 插件支持 SQLite、MySQL 和 PostgreSQL 等数据库[8]，为浮墨的桌面化与本地化实现提供了工程基础。

## 三、方法论 / 项目实施过程

### 1. 需求分析

项目以普通用户的日常生活管理为核心场景，重点覆盖以下需求：

- 快速记录：临时想法、便签、备忘事项。
- 任务管理：待办、子任务、优先级、周期任务。
- 时间管理：倒计时、番茄钟、日历聚合。
- 习惯养成：打卡、连续天数、热力图反馈。
- 智能执行：通过自然语言创建、修改和查询生活事务。
- 数据安全：默认本地保存，减少不必要的数据上传。

### 2. 系统架构

系统分为四层：

- 表现层：Vue 3、Naive UI、Tiptap 富文本编辑器。
- 状态层：Pinia 管理便签、待办、习惯、倒计时、设置等模块状态。
- 数据层：SQLite 存储用户本地数据，统一支持增删改查和回收站机制。
- 桌面能力层：Tauri 提供窗口管理、系统托盘、悬浮窗、原生模糊效果和权限控制。

### 3. 核心功能实现

浮墨完整产品包含智能主对话、悬浮便签、记事本、待办事项、倒计时、番茄钟、习惯打卡、日历视图、全局搜索、回收站和设置中心。用户既可以手动操作，也可以通过 AI 入口完成“创建提醒”“整理便签”“生成今日计划”“查询未完成任务”等操作。

在产品交互上，浮墨强调“看得见的反馈”。例如用户创建提醒后，系统会立即展示已创建的事项、时间和提醒方式；用户删除内容后，项目进入回收站而非立即彻底消失；用户进行专注或习惯打卡后，界面通过进度、连续天数和历史记录反馈行为结果。

## 四、测试 / 项目效果验证方式

### 1. 测试目标

项目验证围绕有效性、效率、满意度、稳定性和隐私信任五类指标展开。

| 指标 | 说明 |
| --- | --- |
| 任务完成率 | 用户能否独立完成指定操作 |
| 平均完成时间 | 完成同一任务所需时间 |
| 操作步数 | 从需求产生到任务落地的点击和输入成本 |
| SUS 可用性评分 | 衡量系统整体易用性 |
| 感知有用性 / 易用性 | 基于技术接受模型设计问卷 |
| 稳定性 | 崩溃、卡顿、数据丢失等异常情况 |
| 隐私信任 | 用户对本地存储和 AI 数据处理方式的接受程度 |

### 2. 测试方案

预计招募 10-20 名受试者，覆盖学生、上班族、自由职业者等典型用户。采用“对照实验 + 问卷调查 + 访谈”的方式进行。

测试任务包括：

1. 用自然语言创建明天的提醒。
2. 创建一条带优先级和子任务的待办。
3. 新建一张悬浮便签并设置颜色。
4. 开始一次番茄钟专注。
5. 创建一个习惯并完成打卡。
6. 在日历中查看本周任务。
7. 使用全局搜索查找历史内容。
8. 删除一条记录并从回收站恢复。

对照组使用传统工具组合完成相同任务，例如系统日历、备忘录、待办应用；实验组使用浮墨完成。测试过程中记录任务完成率、完成时间、操作步数、错误次数和用户主观评价。

### 3. 数据采集与汇总

| 数据类型 | 采集方式 |
| --- | --- |
| 完成率 | 测试观察记录 |
| 完成时间 | 计时器或日志记录 |
| 操作步数 | 屏幕录制或埋点统计 |
| 主观满意度 | SUS 十项量表 |
| 感知价值 | 5 点李克特量表 |
| 问题反馈 | 半结构化访谈 |

### 4. 数据分析方法

数据分析阶段仅对数据本身进行解释，不提前推导结论。完成率用于判断核心功能是否可被用户独立使用；完成时间和操作步数用于衡量效率提升；SUS 评分用于评估整体可用性；感知有用性和感知易用性用于判断用户接受度；访谈内容用于归纳高频痛点和优化方向。

## 五、结论

基于产品定位与验证框架，浮墨的核心价值体现在以下三方面。

第一，浮墨将 AI 从“建议型助手”推进到“执行型生活管家”。用户无需理解复杂软件结构，只需用自然语言表达需求，系统即可转化为具体任务、提醒或记录，降低了生活管理工具的使用门槛。

第二，浮墨通过统一模块解决了个人效率工具割裂的问题。便签、待办、倒计时、番茄钟、习惯和日历不再是孤立功能，而是围绕用户生活事务形成完整闭环。

第三，浮墨采用本地优先的数据策略，回应了用户对 AI 产品隐私和数据控制权的担忧。相比完全云端化的工具，浮墨更适合承载个人生活记录、提醒和习惯数据。

项目仍存在进一步优化空间：一是 AI 意图识别需要持续覆盖更多自然表达；二是提醒策略可进一步结合用户行为，减少不必要打扰；三是未来可扩展多端同步、端侧模型、语音输入和更细粒度的数据权限控制。

后续开发可围绕三条路线继续推进：第一，完善 AI 工具调用体系，使其支持更多跨模块复合任务；第二，建立真实用户测试样本库，用持续数据验证产品体验；第三，探索本地模型与云端模型结合的混合架构，在智能能力和隐私保护之间取得更好平衡。
